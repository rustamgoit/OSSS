# Варіант A — Аналіз життєвого циклу контейнера

## 1. Запуск контейнера

### Команда запуску

```bash
docker run -d --name homework-http -p 8080:8000 python:3.12-slim python -m http.server 8000
```

### Перевірка контейнера

```bash
docker ps
```

### Результат

```text
CONTAINER ID   IMAGE              COMMAND                  STATUS                  PORTS                                         NAMES
250349aefa20   python:3.12-slim   "python -m http.serv…"   Up Less than a second   0.0.0.0:8080->8000/tcp, [::]:8080->8000/tcp   homework-http
```

### Коментар

Контейнер успішно запущений. HTTP-сервер працює всередині контейнера та доступний на порту **8080** хостової машини.

---

# 2. Аналіз процесу

### Перегляд процесу PID 1

```bash
docker exec homework-http sh -c 'cat /proc/1/cmdline | tr "\0" " "'
```

### Результат

```text
python -m http.server 8000
```

### Коментар

Процес із PID 1 є головним процесом контейнера. Саме він запускається командою `docker run` і визначає життєвий цикл контейнера. Поки цей процес працює — контейнер також працює.

---

# 3. Завершення контейнера

### Команда

```bash
docker stop homework-http
```

### Коментар

Під час виконання `docker stop` Docker надсилає головному процесу сигнал **SIGTERM**, який дозволяє програмі коректно завершити роботу.

Якщо процес не реагує на **SIGTERM**, через декілька секунд Docker надсилає сигнал **SIGKILL**, який примусово завершує процес.

---

# 4. Перегляд логів

### Команда

```bash
docker logs homework-http
```

### Коментар

Команда `docker logs` показує інформацію, яку головний процес контейнера записує у потоки **stdout** та **stderr**. Саме ці потоки Docker автоматично зберігає як журнали контейнера.

---

# Висновок

Контейнер працює доти, доки виконується його головний процес (**PID 1**). У цьому завданні таким процесом був Python HTTP Server (`python -m http.server 8000`). Після виконання команди `docker stop` процес завершується, тому завершується і робота контейнера.
